Two Player Game

Main Menu

This is a Two Player Game that I designed and programmed in Android Studio for my Mobile Programming course. This game involves 2 players to be able to play against each other on a single device with individual controls and attempt to blow each other up using the stationary bomb locations.

SRC Code

Starting Battle

Once the START button is clicked, both players are sent to the ARENA and they can begin their battle. Each player is distinguished by their primary color (RED or BLUE) and each have their own controls, platform, bombs, and healthbar.

Movement

There are only two directions a player can move which is LEFT or RIGHT. Both controls can be pressed simultaneously so RED can be mirroring BLUE while BLUE attempts to reach a bomb location and vice versa.

Bomb Damage

Objective of the game is to catch each other within the stationary bomb locations to deal damage. Here we can see an example of each player doing so. Each bomb causes damage on mirror platform and deals 1/3 damage to the player's healthbar.

Game Over

Once a player has reached the end of their healthbar then GAME OVER. The score is recorded and then both players are sent to the GAME OVER screen where they can REMATCH, see SCORES, or return to MAIN MENU.

Scoreboard

A list of previous game battles is recorded within the SCOREBOARD activity so that players can see which player won which battle. They also have the option to reset the SCOREBOARD so that they can start fresh and begin another record.

Communications App

Login Page

This is a Communications Application that I helped design and program in collaboration with 3 other members during our Software Engineering final. This app allows Employees to communicate with each other through virtual servers that our team created. This is the Login Page.

Click to see our Source code!

SRC Code

User Chats

This is the UI design once an Employee logs into their account. They will see their existing coversations along with messages received when the Employee was offline. Here we can see a conversation between two users who are online at the same time.

Storing Employees

Employee has 4 attributes which are name, password, status, and role. Status is used to find if Employee is online or offline. Role determines whether an Employee is an Admin or not. Each Employee is also stored within an ArrayList with all other Employees.

Group Chats

Our app also allows for a user to create group chats of multiple Employees in one conversation. This conversation appears in Employees UI when they receive the first message and then are able to reply as needed.

Creating New Chats

When creating a new chat, a user is given a list of all Employees within the Database to choose from. Each name is either green or orange to indicate whether that Employee is online or offline. Regardless of status, each person within the list can be chosen for a group chat.

Storing New Chats

We stored our messages within a chatLog that had multiple chatLog lists within. Each chatLog list contains a conversation that was created by an Employee. This way all conversations are saved and accounts with role ADMIN have access for later viewing.

Admin Settings

Employees with role ADMIN have access to a settings tab which gives them options such as CREATE, DELETE, and VIEW. This allows the ADMIN to create new Employee accounts, delete existing Employee accounts, and view all chat messages of other Employees that do not have ADMIN role.

Creating a New User

This is the UI when an ADMIN creates a new Employee account. They provide the account name, account password, and decide the role of this Employee. Once created, this account is stored along other Employee accounts and is ready to begin their own conversations.

Multiple Users Online

Our app allows for multiple users to be online at one time which is only possible by our virtual server design. Once the server is online, a ChatService thread is created for each client and then they can run their app on their device.

View All Chatlogs

This is the UI when an ADMIN views all chatLogs that do not contain another ADMIN within them. They receive a list that contains all conversations along with all Employees within that conversation. The ADMIN has access to view all of these at any time.

Example

Here is an example of an ADMIN that is viewing 3 different conversations at one time. There are a combination of normal chats and group chats as well, which ADMIN has access to view.
